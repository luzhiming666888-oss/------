## 第 1 章 问题分析

### 1.1 项目背景

随着高校信息化建设的推进，选课管理系统已成为教务工作的核心组成部分。一所中等规模高校在校学生数千人，每学期开设课程上百门，产生的选课记录数量级在万条以上。目前许多基础管理系统仍使用线性顺序表或简单文件进行存储，随着数据量增长，查询和统计效率急剧下降：按学号/课程精确查找需线性扫描 O(n)，无法满足实时响应需求；成绩统计与多条件筛选缺乏灵活支持；数据持久化不规范，缺乏通用格式。

本项目通过手工实现 AVL 树与哈希表双索引，验证数据结构理论在真实业务场景中的工程价值，同时锻炼 C 语言系统编程能力。系统以"学号\_课程号"为唯一主键，支持精确增删改查、多条件筛选、多关键字排序、成绩统计分析等功能，并配备自动化性能测试模块，在 100/1000/10000 三档数据规模下进行性能对比分析。

### 1.3 数据规模与性能要求

| 测试规模 | 记录条数 | 预期查询响应时间 | 预期加载时间 | 实测查询响应时间 | 实测加载时间 |
| :------: | :------: | :--------------: | :----------: | :--------------: | :----------: |
| 小规模   | 100      | < 1 ms           | < 1 ms       | 哈希 0.056 μs / AVL 0.070 μs | 0.022 ms |
| 中规模   | 1,000    | < 5 ms           | < 5 ms       | 哈希 0.078 μs / AVL 0.084 μs | 0.263 ms |
| 大规模   | 10,000   | < 20 ms          | < 10 ms      | 哈希 0.120 μs / AVL 0.120 μs | 3.659 ms |
| 压力规模 | 50,000   | < 50 ms          | < 100 ms     | 哈希 0.545 μs / AVL 0.275 μs | 34.950 ms |
| 极限规模 | 100,000  | < 100 ms         | < 300 ms     | 哈希 1.794 μs / AVL 0.308 μs | 126.713 ms |

性能边界论证：AVL 树的 O(log n) 在 n=100000 时，log₂100000 ≈ 17 次比较，每次仅为字符串 strcmp 操作，现代 CPU 可在微秒级完成。实测数据显示，100000 条记录批量插入（含双索引构建）仅耗时 126.713 ms，AVL 树实际高度 17（理论最大约 24），验证了理论分析。哈希表在万级以上因固定容量 1021 而严重超载（100K 时装载因子 9794%），查找性能从 0.058 μs 退化到 1.794 μs，但仍远低于毫秒级要求。

空间约束：每条 `CourseRecord` 的内存占用约 240 字节，100000 条约 24 MB，加上 AVL 节点（约 2.7 MB）和哈希节点（约 1.6 MB），总计约 28 MB，仍低于普通 PC 内存限制。

## 第 2 章 数据结构设计

### 2.1 逻辑结构设计

选课记录在逻辑上是一组**无序的扁平键值对集合**。本系统为满足不同操作场景的性能需求，将同一份数据同时映射到两种逻辑结构中，构成**双索引架构**：

- **AVL 平衡二叉搜索树**：以 `key`（学号\_课程号）为排序键，维护所有记录的有序性，支持 O(log n) 的精确增删查改及中序遍历（有序输出）。
- **拉链法哈希表**：以相同的 `key` 为哈希键，提供 O(1) 均摊的精确查找，用于高频精确检索场景。

两种结构共享同一块 `CourseRecord` 堆内存（通过指针引用，不进行数据拷贝），由 AVL 树节点负责内存的唯一所有权与最终释放，哈希表仅持有裸指针。

**唯一主键设计理由**：选择"学号\_课程号"（而非自增 ID）作为主键——它天然具有业务唯一性，无需额外维护自增序列，且字符串排序规则简单明确。主键格式为 `<12位学号>_<8位课程号>`，共 22 字节（含 '\0'）。

**双索引必要性**：AVL 树支持范围查询与有序遍历（用于排序输出、成绩统计等），而哈希表的 O(1) 查找适合高频的单条记录精确检索（如修改成绩时的查找操作）。两者在查询模式上天然互补，共享同一块 `CourseRecord` 堆内存，不产生数据冗余。

双索引架构示意如下：

```
+-----------+         +------------------+
|  AVL Tree |──data──▶|  CourseRecord    |◀──data──+  HashTable
|  (有序)   |         |  (唯一堆内存)    |         |  (快速检索)
+-----------+         +------------------+         +-----------+
```

### 2.2 ADT 定义

#### 2.2.1 CourseRecord（选课记录）

```c
/* 字段长度宏 */
#define STUDENT_ID_LEN   13   /* 学号 12 位 + '\0' */
#define NAME_LEN         32   /* 姓名（含中文 UTF-8）*/
#define COLLEGE_LEN      64   /* 学院名称 */
#define COURSE_ID_LEN    9    /* 课程编号 8 位 + '\0' */
#define COURSE_NAME_LEN  64   /* 课程名称 */
#define SEMESTER_LEN     16   /* 学期，格式 "2024-2025-1" */
#define DATE_LEN         11   /* 日期，格式 "YYYY-MM-DD" */
#define KEY_LEN          (STUDENT_ID_LEN + COURSE_ID_LEN + 1)
                              /* 主键 = 学号(12) + '_' + 课程号(8) + '\0' = 22 */

typedef struct CourseRecord {
    char  student_id[STUDENT_ID_LEN];   /* 学号（12位）         */
    char  name[NAME_LEN];               /* 学生姓名             */
    char  college[COLLEGE_LEN];         /* 所属学院             */
    char  course_id[COURSE_ID_LEN];     /* 课程编号（8位）      */
    char  course_name[COURSE_NAME_LEN]; /* 课程名称             */
    float credit;                       /* 学分                 */
    char  semester[SEMESTER_LEN];       /* 选课学期             */
    char  date[DATE_LEN];               /* 选课日期             */
    int   score;                        /* 成绩（0-100，-1未录）*/
    char  key[KEY_LEN];                 /* 唯一主键（自动生成） */
} CourseRecord;
```

#### 2.2.3 HashTable（拉链法哈希表）

```c
typedef struct HashNode {
    CourseRecord    *data;  /* 记录指针（非所有权）*/
    struct HashNode *next;  /* 拉链下一节点        */
} HashNode;

typedef struct HashTable {
    HashNode **buckets;     /* 槽位数组            */
    int        capacity;    /* 槽位总数（质数）    */
    int        size;        /* 当前记录总数        */
} HashTable;
```

**哈希表 ADT 操作集合：**

| 操作名称   | 原型                                                              | 均摊复杂度 |
| ---------- | ----------------------------------------------------------------- | :--------: |
| 初始化     | `int hash_init(HashTable *ht, int capacity)`                      | O(m)       |
| 插入       | `int hash_insert(HashTable *ht, CourseRecord *record)`            | O(1)       |
| 删除       | `int hash_delete(HashTable *ht, const char *key)`                 | O(1)       |
| 精确查找   | `CourseRecord *hash_search(HashTable *ht, const char *key)`       | O(1)       |
| 更新指针   | `int hash_update(HashTable *ht, const char *key, CourseRecord *r)`| O(1)       |
| 状态统计   | `void hash_print_stats(const HashTable *ht)`                      | O(m)       |
| 销毁       | `void hash_destroy(HashTable *ht)`                                | O(m+n)     |

> *m = 槽位数（默认 1021），n = 记录数*

**哈希函数说明**：本项目采用 FNV-1a（Fowler-Noll-Vo 1a）哈希算法，其核心步骤为 `hash ^= byte; hash *= 16777619`，初始偏移量 `2166136261`。相比备选的 DJB2 算法（`hash * 33 ^ c`，初始值 5381），FNV-1a 先 XOR 后乘的策略能更好地混淆高低位，在短字符串场景下碰撞率更低、分布更均匀，尤其适合本项目 22 字符长度的主键（学号\_课程号）。

**平衡因子定义**：AVL 树平衡因子 BF = 左子高 - 右子高，BF ∈ {-1, 0, 1} 为合法状态，|BF| = 2 时触发旋转修复。

**内存所有权说明**：`CourseRecord` 堆内存唯一所有权在 AVL 树节点，哈希表的 `HashNode.data` 为裸指针共享，`hash_destroy()` 仅释放 `HashNode` 链表节点本身，不 `free(data)`，防止双重释放错误。AVL 删除采用"前驱替换"策略时，前驱节点的 data 指针转移后置 NULL，防止递归删除时双重释放。

### 2.3 存储结构方案

#### 2.3.1 内存分配方式

本系统采用**动态堆分配**策略：

- 每条 `CourseRecord` 在插入时通过 `calloc(1, sizeof(CourseRecord))` 分配，生命周期由对应 AVL 节点管理。
- 每个 `AVLNode` 同样通过 `malloc(sizeof(AVLNode))` 分配，`avl_destroy()` 负责后序遍历释放。
- `HashNode` 链表节点通过 `malloc(sizeof(HashNode))` 分配，`hash_destroy()` 负责逐槽释放。
- 哈希表槽位数组通过 `calloc(capacity, sizeof(HashNode*))` 分配并清零。

#### 2.3.2 空间开销估算

| 结构         | 单节点字节数                          | n = 10,000 时总占用    |
| ------------ | ------------------------------------- | ---------------------- |
| CourseRecord | ≈ 13+32+64+9+64+4+16+11+4+22 = 239 B，对齐后约 **240 B** | ≈ 2.4 MB |
| AVLNode      | 8（指针）+ 4（height）+ 8+8（left/right）= **28 B**（64位系统） | ≈ 0.27 MB |
| HashNode     | 8（data指针）+ 8（next指针）= **16 B** | ≈ 0.15 MB |
| HashTable buckets 数组 | 1021 × 8 B = 约 **8 KB**   | 固定                   |
| **合计**     |                                       | **≈ 2.83 MB**          |

#### 2.3.3 文件持久化格式

数据文件采用 **JSON Lines** 格式（每行一个独立 JSON 对象），不依赖任何外部解析库，序列化与反序列化均为手写实现。示例行如下：

```json
{"sid":"202401011001","name":"张三","college":"软件学院","cid":"CS300101","cname":"数据结构","credit":3.0,"semester":"2024-2025-1","date":"2024-09-01","score":81}
```

**为何选 JSON Lines 而非 CSV**：JSON Lines 自描述性更强，字段顺序无关，新增字段时对旧文件有良好的前向兼容性；CSV 在含逗号的字段（如课程名称含英文逗号）时需要额外的转义处理，增加解析复杂度。

**手写解析器的核心思路**：扫描字符串，找到 `"field"` 模式后提取引号内的值（字符串字段）或直接读取数字（数值字段），遇到非法格式返回 -1 报错。支持 `\"` 转义引号处理。

**数据文件路径约定**：运行目录下的 `data/records.jsonl`，程序启动时自动创建（若不存在）。

## 第 3 章 方案对比与选型论证

### 3.1 候选方案介绍

在进行技术选型时，本项目重点考察了以下三种数据结构方案：

**方案一：双向链表（Doubly Linked List）**

双向链表是最直观的线性动态结构，插入和删除操作在已知节点位置时为 O(1)，但由于缺乏索引，按键查找必须从头遍历，时间复杂度为 O(n)，在大规模数据场景下性能不可接受。

**方案二：哈希表（Hash Table，拉链法）**

哈希表通过散列函数将键映射到桶槽，实现均摊 O(1) 的精确查找。其缺点是：① 无法保持键的有序性，不支持范围查询和有序遍历；② 哈希碰撞严重时性能退化；③ 动态扩容实现复杂。

**方案三：AVL 平衡二叉搜索树（AVL Tree）**

AVL 树是一种自平衡的 BST，任意节点左右子树高度差不超过 1，保证最坏情况下增删查均为 O(log n)。相较于普通 BST，AVL 树避免了退化为链表的极端情况；相较于红黑树，AVL 树的平衡条件更严格，查询性能略优（树高更低），但旋转次数略多。

---

### 3.2 多维度对比分析

| 对比维度           | 双向链表      | 哈希表（拉链）         | AVL 树              |
| ------------------ | :-----------: | :--------------------: | :-----------------: |
| **精确查找**       | O(n)          | O(1) 均摊              | O(log n)            |
| **插入**           | O(1)（尾插） | O(1) 均摊              | O(log n)            |
| **删除**           | O(n)（需查找）| O(1) 均摊              | O(log n)            |
| **有序遍历**       | O(n)（有序需额外排序）| ✗ 不支持原生有序 | O(n)（中序天然有序）|
| **范围查询**       | O(n)          | ✗ 不支持               | O(log n + k)        |
| **最坏情况查找**   | O(n)          | O(n)（退化时）         | O(log n)            |
| **空间开销**       | 低            | 中（桶数组 + 链表节点）| 中（节点含 height） |
| **实现复杂度**     | 低            | 中                     | 高（旋转逻辑复杂）  |
| **内存局部性**     | 差            | 一般                   | 一般                |
| **适用主要场景**   | 有序链式操作  | 高频精确检索           | 增删查改 + 有序输出 |

从表中可见，单一哈希表虽然精确查找最快，但无法支持有序遍历和范围查询；而 AVL 树在各操作均能保证 O(log n) 的稳定性能，且天然支持有序输出。两者各有侧重，因此本项目采用双索引架构，以哈希表承担高频精确检索，以 AVL 树承担有序遍历与范围操作。实测数据（见 5.3 节）验证了这一设计：在 10000 条记录下，哈希表查找 0.120 μs/op，AVL 树查找 0.120 μs/op，两者均在微秒级完成，而 AVL 树额外提供了有序遍历能力。



---

### 3.3 选型结论

综合 3.2 节的对比分析，本项目最终采用 **"AVL 树 + 哈希表"双索引架构**，并使用**手写极简 JSON 序列化/反序列化**方案，工程合理性论证如下：

**（1）双索引优势互补**

AVL 树与哈希表在查询模式上天然互补：哈希表以 O(1) 均摊复杂度处理高频精确查找（如"按学号\_课程号检索单条记录"），AVL 树以 O(n) 中序遍历支持全集有序输出（如"按成绩排序展示所有记录"）和 O(log n) 的增删维护。两者共享同一块 `CourseRecord` 堆内存，不产生数据冗余。

**（2）AVL 树优于普通 BST 与跳表**

普通 BST 在数据按主键顺序插入时退化为链表，查询退化为 O(n)，在校园系统中学号通常按序递增，此退化场景极易触发。跳表虽然实现相对简单，但随机层数引入不确定性，且在 C 语言中需要手动管理多层指针，维护成本高于 AVL 树。

**（3）手写 JSON 满足工程约束**

课程设计明确禁止使用第三方库（`cJSON`、`jansson` 等），而 JSON Lines 格式每行结构固定、字段有限，手写扫描解析器代码量约 80～120 行，完全可控。相较于自定义二进制格式，JSON Lines 具有人类可读性，便于调试和数据导入/导出。

### 3.4 复杂度分析

#### AVL 树各操作复杂度

| 操作       | 最优         | 平均         | 最坏         | 空间     |
| ---------- | :----------: | :----------: | :----------: | :------: |
| 查找       | O(1)（根节点）| O(log n)    | O(log n)     | O(1)     |
| 插入       | O(log n)     | O(log n)     | O(log n)     | O(log n)（栈） |
| 删除       | O(log n)     | O(log n)     | O(log n)     | O(log n)（栈） |
| 中序遍历   | O(n)         | O(n)         | O(n)         | O(log n)（栈） |
| 旋转（单次）| O(1)        | O(1)         | O(1)         | O(1)     |
| 树高上界   | —            | —            | ≤ 1.44 log₂(n+2) | —   |

#### 哈希表各操作复杂度

| 操作     | 最优   | 平均（装载因子 α < 0.75）| 最坏（全碰撞）| 空间   |
| -------- | :----: | :----------------------: | :-----------: | :----: |
| 查找     | O(1)   | O(1 + α) ≈ O(1)          | O(n)          | O(1)   |
| 插入     | O(1)   | O(1)                     | O(n)          | O(1)   |
| 删除     | O(1)   | O(1)                     | O(n)          | O(1)   |
| 初始化   | O(m)   | O(m)                     | O(m)          | O(m)   |

> *m = 槽位数（本项目默认 1021），α = n/m 为装载因子*

**AVL 树高公式推导**：AVL 树高 h ≤ 1.44 log₂(n+2) 的直观来源是最坏情况为 Fibonacci 树，此时节点数 n(h) = n(h-1) + n(h-2) + 1，解出反函数即为上界。实测数据验证：100 条记录树高 7（理论最大 10），1000 条树高 10（理论最大 14），10000 条树高 14（理论最大 19），均低于理论上界。

**FNV-1a 散列均匀性**：本项目主键格式为"12位学号\_8位课程号"，字符多样性高，FNV-1a 碰撞概率极低。实测在 100 条记录时装载因子 9.79%（最长链 2），1000 条时 97.94%（最长链 7），说明 FNV-1a 对该主键格式的散列分布均匀性良好。10000 条时因容量固定 1021，装载因子达 979.43%（最长链 23），提示大容量数据需要动态扩容机制（见 6.3 节改进方向）。

### 4.2 核心函数说明

#### 4.2.1 哈希精确检索：`hash_search()`

**函数原型：**

```c
CourseRecord *hash_search(HashTable *ht, const char *key);
```

**功能描述：**

在哈希表中按唯一主键（学号\_课程号）进行 O(1) 均摊的精确查找。调用者传入 `HashTable` 指针和 key 字符串，函数返回匹配的 `CourseRecord` 指针（未找到返回 NULL）。该函数是双索引架构中"高频精确检索"的主要入口，例如修改记录前先用哈希表快速定位目标，再通过 AVL 树维护有序性。

**实现思路：**

1. 调用 `hash_fnv1a(key)` 计算散列值，对 `ht->capacity` 取模得到槽位下标 `idx`。
2. 从 `ht->buckets[idx]` 开始遍历拉链，逐节点比较 `node->data->key` 与 `key`。
3. 找到则返回 `node->data`，遍历完未找到则返回 `NULL`。

**时间复杂度：** 均摊 O(1)，最坏 O(n)（全槽碰撞）

---

#### 4.2.2 AVL 树插入与旋转：`avl_insert()` / `avl_rotate_*()`

**函数原型：**

```c
AVLNode *avl_insert(AVLNode *root, CourseRecord *record);
AVLNode *avl_rotate_ll(AVLNode *z);   /* LL 型右旋 */
AVLNode *avl_rotate_rr(AVLNode *z);   /* RR 型左旋 */
AVLNode *avl_rotate_lr(AVLNode *z);   /* LR 型双旋 */
AVLNode *avl_rotate_rl(AVLNode *z);   /* RL 型双旋 */
```

**插入流程说明：**

1. 按 BST 规则递归向下定位插入位置，`strcmp` 比较新记录的 `key` 与当前节点 `data->key`，小于则走向左子树，大于则走向右子树，等于则拒绝插入（主键唯一性约束）。
2. 递归到空位时，调用 `avl_new_node(record)` 创建新叶节点（`height = 1`），挂接到父节点的对应子指针上。
3. 递归返回过程中，逐层调用 `AVL_UPDATE_HEIGHT(node)` 宏更新高度（取左右子树最大高度 + 1）。
4. 计算当前节点平衡因子 BF，若 |BF| = 2 则判断失衡类型并选择对应旋转：
   - BF = 2 且左子 BF ≥ 0 → LL 型（右旋 `avl_rotate_ll`）
   - BF = 2 且左子 BF < 0 → LR 型（先对左子左旋，再对当前节点右旋 `avl_rotate_lr`）
   - BF = -2 且右子 BF ≤ 0 → RR 型（左旋 `avl_rotate_rr`）
   - BF = -2 且右子 BF > 0 → RL 型（先对右子右旋，再对当前节点左旋 `avl_rotate_rl`）
5. 旋转后更新涉及节点的高度，返回新子树根节点。

**删除流程说明：**

1. 按 BST 规则递归查找待删除节点，`strcmp` 比较 `key` 定位目标。
2. 找到目标节点后分三种情况处理：
   - **叶子节点（无子节点）**：直接 `free(data)` 和 `free(node)`，返回 NULL。
   - **单子节点**：用子节点替换当前节点，释放当前节点的 `data` 和 `node` 本身，返回子节点指针。
   - **双子节点**：采用"前驱替换"策略——找到左子树的最大节点（前驱），保存前驱的 key（因 data 转移后无法再读取），释放当前节点的旧 data，将前驱的 data 指针转移给当前节点，然后将前驱节点的 data 置 NULL（防止双重释放），最后递归删除前驱节点（此时前驱至多有一个左子）。
3. 递归返回过程中，同样逐层更新高度并执行 `avl_rebalance` 平衡修复。

> 建议手绘 LL、RR 旋转的前后示意图各一张，贴在此处。

#### 4.2.3 多条件筛选：`filter_records()`

**函数原型：**

```c
int filter_records(CourseRecord **in,  int in_cnt,
                   CourseRecord **out, const FilterCriteria *fc);
```

**筛选条件结构：**

```c
typedef struct FilterCriteria {
    char  student_id[STUDENT_ID_LEN]; /* 按学号前缀/精确匹配；空字符串跳过  */
    char  college[COLLEGE_LEN];       /* 按学院精确匹配；空字符串跳过        */
    char  course_id[COURSE_ID_LEN];   /* 按课程编号精确匹配；空字符串跳过    */
    char  semester[SEMESTER_LEN];     /* 按学期精确匹配；空字符串跳过        */
    int   score_min;                  /* 成绩下限（-1 表示不设下限）         */
    int   score_max;                  /* 成绩上限（-1 表示不设上限）         */
    float credit_min;                 /* 学分下限（< 0 表示不设下限）        */
} FilterCriteria;
```

**功能描述：** 从输入记录指针数组中，逐条检查每条记录是否满足 `FilterCriteria` 中所有非空字段条件，满足全部条件的记录指针写入 `out` 数组，返回满足条件的记录总数。

**实现思路：**

1. 遍历 `in` 数组中的每条记录指针。
2. 对每条记录，依次检查 `fc` 中的各字段：
   - `student_id` 非空 → 要求记录的学号与之前缀匹配或精确匹配
   - `college` 非空 → 要求学院名称精确匹配（`strcmp == 0`）
   - `course_id` 非空 → 要求课程编号精确匹配
   - `semester` 非空 → 要求学期字符串精确匹配
   - `score_min >= 0` → 要求成绩 ≥ 下限（`score == -1` 的未录入项不满足任何成绩条件）
   - `score_max >= 0` → 要求成绩 ≤ 上限
   - `credit_min >= 0` → 要求学分 ≥ 下限
3. 所有非空条件均满足时，将记录指针写入 `out[out_cnt++]`。
4. 返回 `out_cnt`。

#### 4.2.4 性能测试：`benchmark_run_all()`

**函数原型：**

```c
void benchmark_run_all(void);
```

**功能描述：**

自动生成 100/1000/10000 三档测试数据，使用 Windows `QueryPerformanceCounter` 实现微秒级高精度计时，对 AVL 树和哈希表进行全方位性能对比。测试项目包括：批量插入耗时（含双索引构建）、AVL 树实际高度 vs 理论最大高度、哈希表与 AVL 树精确查找耗时对比、哈希表运行状态统计（装载因子、链长分布）。报告同时输出到终端和 `data/benchmark_report.txt` 文件。

5. **哈希状态统计**：遍历所有槽位，统计最长链、平均链长、空闲槽数、装载因子。
6. **报告输出**：使用 `print_table_line()` 和 `print_table_row()` 绘制 ASCII 表格，同时写入文件。

#### 4.2.5 压力测试：`stress_test_run_all()`

**函数原型：**

```c
void stress_test_run_all(void);
```

**功能描述：**

自动在 1K / 5K / 10K / 50K / 100K 五档数据规模下，对 AVL 树和哈希表进行插入、查找、删除全方位性能测试。每档分独立两轮运行（轮A测插入+查找，轮B测插入+删除），互不干扰。测试完成后自动生成 `data/stress_report.html` 可视化报告（Chart.js 5 个图表 + 汇总表 + 分析结论）。

1. **数据生成**：`stress_gen_records()` 按固定随机种子（srand(42)）生成指定数量的 `CourseRecord`，学号采用分段策略（`202401%02d%04d`）确保 12 位学号在 `STUDENT_ID_LEN=13` 缓冲区限制内不截断。
2. **独立轮次**：轮A 生成→插入→查找（min(N, 50000) 次随机查询）→统计收集→释放；轮B 重新生成→插入→随机删除（min(N, 10000) 条）计时→释放。
3. **删除 key 安全拷贝**：删除测试中，先将随机选中的 key 拷贝到 `char (*)[KEY_LEN]` 数组中（值拷贝而非指针），避免 AVL 删除释放 CourseRecord 后 key 指针悬空。
4. **HTML 报告生成**：使用 Chart.js CDN 绘制 5 个图表（插入耗时趋势、查找性能对比、删除耗时趋势、哈希装载因子/链长、AVL 树高对比），内嵌 CSS 实现 LeetCode/掘金风格极简 UI。

#### 4.2.6 外部排序：`external_sort_run()`

**函数原型：**

```c
void external_sort_run(int total_records, int mem_limit, ExternalSortStats *stats);
```

**功能描述：**

模拟内存仅能容纳 `mem_limit`（默认 1000）条记录的极端场景，对 `total_records`（默认 100000）条选课记录按学号进行外部多路归并排序。全程输出中间过程信息（每轮归并的读写次数、临时文件数量等），最终输出 `data/external_sort_output.jsonl`。

**实现思路：**

1. **Phase 1 — 分片排序**：随机生成 100K 条测试数据写入 `data/external_sort_input.jsonl`；从输入文件逐行读取，每满 1000 条在内存中用快速排序（按 `student_id`），写入临时 chunk 文件 `tmp/chunk_XXXX_0.jsonl`。100K 条共产生 100 个 chunk。
2. **Phase 2 — 多路归并**：使用 k=20 路最小堆（`HeapNode` 结构体含 `CourseRecord` 值拷贝 + 来源 chunk 编号）。每轮打开 k 个 chunk 文件，堆顶弹出最小记录写入输出文件，从对应 chunk 补充下一条。Pass 1 将 100 个 chunk 归并为 5 个，Pass 2 将 5 个归并为 1 个最终输出。
3. **I/O 统计**：全程追踪 `total_reads`、`total_writes`、峰值临时文件数、各轮归并详情，写入 `ExternalSortStats` 结构体。
4. **临时文件清理**：归并完成后自动删除中间 chunk 文件，`tmp/` 目录下的最终输出保留。

#### 4.3.1 FNV-1a 哈希函数

```c
unsigned long hash_fnv1a(const char *key) {
    unsigned long hash = 2166136261UL;
    const unsigned long FNV_PRIME = 16777619UL;
    while (*key) {
        hash ^= (unsigned char)*key++;
        hash *= FNV_PRIME;
    }
    return hash;
}
```

FNV-1a 算法由 Glenn Fowler、Landon Curt Noll 和 Phong Vo 共同设计。其核心思想是对每个字节先执行 XOR 异或操作再乘以质数 `16777619`（FNV prime），初始偏移量 `2166136261`（FNV offset basis）是经验值。相比 DJB2 的 `hash * 33 ^ c`，FNV-1a 先 XOR 后乘的策略能更好地混淆高低位，在短字符串场景下碰撞率更低、分布更均匀，尤其适合本项目 22 字符长度的主键（学号\_课程号）。

#### 4.3.2 AVL 旋转核心代码

以 LL 型右旋为例，展示指针操作与高度更新的完整实现：

```c
/**
 * @brief   LL 型失衡修复 —— 右旋
 * z 的左子树的左子树插入导致失衡。
 * 将 z 的左子 l 提升为新子树根，z 成为 l 的右子。
 *
 * 旋转前:        z               旋转后:      l
 *               / \                         /   \
 *              l   T4                     T1     z
 *             / \                                / \
 *            T1  T2                            T2  T4
 */
AVLNode *avl_rotate_ll(AVLNode *z) {
    AVLNode *l = z->left;     /* l 为 z 的左子节点，旋转后成为新根 */

    z->left   = l->right;     /* l 的右子树 T2 挂到 z 的左侧 */
    l->right  = z;            /* z 成为 l 的右子 */

    AVL_UPDATE_HEIGHT(z);     /* 先更新下层节点 z 的高度 */
    AVL_UPDATE_HEIGHT(l);     /* 再更新新根 l 的高度 */

    return l;                 /* 返回新子树根 */
}
```

**关键点分析：**

1. **指针操作顺序**：必须先将 `l->right` 赋给 `z->left`（保存 T2 子树），再将 `z` 赋给 `l->right`，顺序不可颠倒，否则 `l->right` 原有的引用会丢失。
2. **高度更新顺序**：先更新处于下层的 `z`（其子树已确定），再更新上层的 `l`（依赖 `z` 的新高度），保证 `AVL_UPDATE_HEIGHT` 宏读取到的子树高度是最新的。
3. **平衡修复函数 `avl_rebalance`**：在插入/删除的递归回溯路径上，对每个节点计算 `avl_balance_factor`，根据 BF 值和子节点 BF 的符号判断 LL/RR/LR/RL 四种失衡类型，调用对应旋转函数。

#### 4.3.3 JSON Lines 反序列化核心代码

```c
/**
 * @brief  在 json 字符串中查找 "field" 后面的值，写入 out_buf
 * @return 0 成功；-1 未找到该字段
 *
 * 支持的值类型：字符串 ("...") 和整数/浮点数（裸数字）
 */
static int json_extract(const char *json, const char *field,
                        char *out_buf, int out_sz) {
    /* 构造搜索模式："field" */
    char pattern[64];
    snprintf(pattern, sizeof(pattern), "\"%s\"", field);

    const char *p = strstr(json, pattern);
    if (!p) return -1;

    /* 跳过 "field" 部分 */
    p += strlen(pattern);

    /* 跳过空白和冒号 */
    while (*p && (isspace((unsigned char)*p) || *p == ':')) p++;
    if (!*p) return -1;

    if (*p == '"') {
        /* 字符串值：提取到下一个未转义的 '"' */
        p++;
        int i = 0;
        while (*p && *p != '"' && i < out_sz - 1) {
            if (*p == '\\' && *(p + 1) == '"') {
                if (i < out_sz - 1) out_buf[i++] = '"';
                p += 2;
            } else {
                out_buf[i++] = *p++;
            }
        }
        out_buf[i] = '\0';
        return 0;
    } else {
        /* 数值类型：提取连续的数字、小数点、负号 */
        int i = 0;
        while (*p && !isspace((unsigned char)*p) && *p != ',' && *p != '}' && i < out_sz - 1) {
            out_buf[i++] = *p++;
        }
        out_buf[i] = '\0';
        return 0;
    }
}

int json_to_record(const char *json, CourseRecord *out) {
    if (!json || !out) return -1;
    memset(out, 0, sizeof(CourseRecord));

    char buf[256];

    /* 逐字段提取，任一字段缺失则返回 -1 */
    if (json_extract(json, "sid", buf, sizeof(buf)) != 0) return -1;
    util_strncpy(out->student_id, buf, STUDENT_ID_LEN);

    if (json_extract(json, "name", buf, sizeof(buf)) != 0) return -1;
    util_strncpy(out->name, buf, NAME_LEN);
    /* ... 其余字段同理 ... */

    /* 自动生成唯一主键 */
    record_make_key(out);
    return 0;
}
```

**关键点分析：**

1. **字段顺序无关**：`json_extract` 通过 `strstr` 搜索 `"field"` 模式定位字段，不依赖 JSON 中键值对的顺序，符合 JSON 规范。
2. **转义处理**：字符串值中支持 `\"` 转义引号，单元测试已验证 `O"Brien` 等含转义引号的姓名能正确还原。
3. **数值与字符串分流**：通过判断值的首字符是否为 `"` 区分字符串字段与数值字段，分别走不同的提取分支。
4. **安全截断**：所有字段提取均通过 `util_strncpy` 写入目标缓冲区，保证以 `\0` 结尾，防止缓冲区溢出。

#### 4.3.4 CLI 菜单交互核心代码

`main.c` 采用**统一 `fgets` 输入策略**，避免 `scanf` 与 `fgets` 混用导致的 stdin 缓冲区错位问题：

```c
/**
 * @brief  读取一行字符串（去除首尾空白，自动消耗溢出字符）
 */
static void read_line(char *buf, int size) {
    if (!fgets(buf, size, stdin)) {
        buf[0] = '\0';
        return;
    }

    /* 检查 fgets 是否读到了完整的一行（以 \n 结尾） */
    int len = (int)strlen(buf);
    int has_newline = (len > 0 && buf[len - 1] == '\n');

    /* 如果没读到换行符，说明该行还有剩余字符在 stdin 中，
       需要消耗掉，避免污染下一次读取 */
    if (!has_newline) {
        int c;
        while ((c = getchar()) != '\n' && c != EOF) { }
    }

    /* 去除首尾空白（包括 \r\n） */
    util_trim(buf);
}
```

**主菜单循环结构：**

```c
static void menu_main(void) {
```

**关键设计决策：**

1. **统一 `fgets` + `atoi`/`atof`**：所有输入均通过 `read_line` 读取字符串后转换，彻底消除 `scanf` 留残 `\n` 导致 `fgets` 读到空行的经典 C 语言 bug。
2. **缓冲区溢出保护**：当 `fgets` 的 `size` 小于输入行长度时（如 `STUDENT_ID_LEN=13` 读 12 字符学号），`read_line` 自动消耗 stdin 中剩余字符，保证下一次读取对齐。
3. **EOF 优雅退出**：主循环检测 `feof(stdin)`，在管道输入耗尽时自动退出，避免无限循环。
4. **修改操作的原地更新策略**：修改记录时仅更新 `CourseRecord` 字段值（学号和课程编号不可改），由于 AVL 树和哈希表都持有同一指针，无需重建索引。
5. **命令行参数 `--benchmark` / `--stress` / `--extsort`**：`main()` 检测 `argv[1]` 为对应参数时直接运行对应测试模块，不进入交互菜单，便于自动化测试。
6. **Windows UTF-8 编码**：`main()` 开头调用 `SetConsoleOutputCP(65001)` 和 `SetConsoleCP(65001)`，确保终端中文正常显示。

#### 4.3.5 性能测试高精度计时器

```c
/* Windows QueryPerformanceCounter 高精度计时器（微秒级精度）*/
typedef struct {
    LARGE_INTEGER start;
    LARGE_INTEGER freq;
} BenchTimer;

static void timer_start(BenchTimer *t) {
    QueryPerformanceFrequency(&t->freq);
    QueryPerformanceCounter(&t->start);
}

static double timer_elapsed_ms(BenchTimer *t) {
    LARGE_INTEGER end;
    QueryPerformanceCounter(&end);
    return (double)(end.QuadPart - t->start.QuadPart) * 1000.0
           / (double)t->freq.QuadPart;
}
```

**关键点分析：**

1. **为何选择 QueryPerformanceCounter 而非 `clock()`**：`clock()` 精度仅毫秒级，在 100 条记录的微秒级操作中分辨率不足；`QueryPerformanceCounter` 基于 CPU 时钟计数器，精度可达微秒甚至纳秒级，能准确测量单次查找耗时。
2. **频率归一化**：`QueryPerformanceFrequency` 获取时钟频率（通常与 CPU 主频相关），通过 `elapsed = (end - start) * 1000 / freq` 将计数差转换为毫秒。
3. **查找耗时计算**：对 N 次查找总耗时取平均，`hash_us = (total_ms * 1000) / n_queries`，得到单次操作的微秒级耗时。

> 查找测试方法：随机选取已插入的 key，各搜索 min(N, 10000) 次取平均耗时。哈希表 O(1) 均摊查找耗时随规模增长缓慢（0.056→0.120 μs，约 2 倍），主要来自缓存未命中和拉链遍历开销；AVL 树 O(log n) 查找随规模增长而缓慢增加（0.070→0.120 μs，约 1.7 倍），与 log₂(10000/100) ≈ 6.6 的理论比值一致。在 100 条规模下哈希表比 AVL 快 1.25 倍，但随着规模增大两者趋于接近，原因是微秒级操作中固定开销（函数调用、指针解引用）占比增大。

### 6.2 收获与反思

**主题一：手写底层算法的体会**

AVL 树旋转的调试过程是本项目最大的挑战。指针操作极易出错——left/right 搞反、高度未及时更新、旋转后未返回新根等问题频繁出现。通过在 `avl_insert` 和 `avl_delete` 的递归回溯路径上打印平衡因子和树结构，逐步定位了 LR 型旋转中"先对左子左旋"的顺序错误。删除操作的"前驱替换"策略中，最棘手的是防止双重释放：前驱节点的 data 指针转移给目标节点后，必须将前驱的 data 置 NULL，否则递归删除前驱时会再次 free 同一块内存。最终通过保存前驱 key（因 data 转移后无法再读取）的方式解决了这一问题。

**主题二：手写极简 JSON 解析器的体会**

手写字符扫描解析的核心挑战在于正确处理嵌套引号和字段顺序无关性。`json_extract` 通过 `strstr` 搜索 `"field"` 模式定位字段，不依赖 JSON 键值对顺序，符合 JSON 规范。转义引号 `\"` 的处理需要额外的前瞻逻辑——遇到 `\` 时检查下一字符是否为 `"`，若是则写入字面引号并跳过两字符。单元测试中 `O"Brien` 这一用例有效验证了该逻辑的正确性。与直接使用 `cJSON` 库相比，手写解析器虽然工作量更大，但彻底理解了序列化协议的底层原理，也锻炼了 C 语言字符串操作能力。

**主题三：性能测试模块的设计体会**

实现 `benchmark.c` 时，最初使用 `clock()` 函数计时，但在 100 条记录的微秒级操作中分辨率不足（仅能测到 0 ms）。改用 Windows `QueryPerformanceCounter` 后精度提升至微秒级，才得以准确测量单次查找耗时。测试数据生成需与 `gen_test_data.py` 保持相同的数据池和随机种子（srand(42)），保证结果可复现。10000 条数据下哈希表装载因子达 979% 的实测结果，直观揭示了固定容量哈希表的瓶颈，为改进方向提供了数据支撑。

**主题四：压力测试中发现的 AVL 删除 Bug**

在开发 `stress_test.c` 时，系统在 1000 条规模的删除测试中崩溃。通过逐层加 `fflush(stdout)` 调试输出定位到 `avl_delete` 函数。问题出在前驱替换策略：当删除双子节点时，前驱节点的 `data` 指针被转移给目标节点后置为 NULL，但递归删除前驱时 `strcmp(key, root->data->key)` 访问了 NULL 指针导致段错误。此前该 bug 未暴露是因为主程序数据量小、删除操作少，恰好没有触发双子节点删除场景。修复方案是在 `avl_delete` 入口增加 `if (root->data == NULL)` 守卫检查，将 NULL data 的节点按叶子/单子处理。该经历深刻体会到：边界条件测试必须覆盖极端数据规模，因为小规模测试可能恰好绕过了代码的隐蔽缺陷。

**主题五：外部排序中最小堆与文件 I/O 的配合**

实现多路归并时，核心挑战在于如何在 20 个打开的文件流之间高效选出最小值。最小堆的每个节点需要存储 `CourseRecord` 的值拷贝（而非指针），因为堆排序过程中记录会频繁交换位置。每次堆顶弹出后，从对应 chunk 文件读取下一条记录补充到堆中。当某个 chunk 文件读空后，堆的有效大小减 1，继续归并剩余 chunk。最终 100 个初始 chunk 经 2 轮归并（k=20）合并为 1 个有序输出，实测 I/O 总量（600,000 条次）与理论公式 2N × (1 + ⌈log_k(N/M)⌉) 完全吻合，验证了算法的正确性。



---

### 6.3 可改进之处

1. **哈希表动态扩容（rehash）**：当前哈希表容量固定为 1021，实测 10000 条数据时装载因子达 979%，最长链 23，性能退化明显。可实现 rehash 扩容机制——当装载因子超过 0.75 阈值时，分配更大的质数容量（如 2039/4093/8191），重新计算所有 key 的槽位并迁移数据，将大容量下的查找性能恢复到 O(1) 均摊。

2. **红黑树替代 AVL 树**：红黑树的旋转次数更少（平均插入旋转约 1 次 vs AVL 的 2 次），在写密集场景下性能更优。AVL 树的严格平衡条件虽然查询略快，但在频繁增删的场景下红黑树的整体效率更高。

3. **内存池**：为 `AVLNode` 和 `HashNode` 引入简单的内存池，减少 `malloc/free` 碎片化开销。预分配一大块内存，按固定大小切分供节点使用，可显著提升批量插入性能。

4. **多文件并发读写**：当前持久化为单文件全量写入，可改为 WAL（Write-Ahead Log）或增量追加模式，避免退出时全量写入的开销，同时支持崩溃恢复。

5. **图形界面**：将 CLI 替换为基于 ncurses 的 TUI 界面，或使用 Win32 API/Qt 开发 GUI 版本，提升用户交互体验。

### C.4 分析结论

1. **插入性能**：100K 插入耗时约 127ms，增长趋势符合 O(n log n) 理论预期，未出现退化。
2. **查找性能**：哈希表查找在 100K 时退化到 1.8μs（链长 102），但仍优于 AVL 的 0.3μs？这里出现了反直觉现象——AVL 查找在 50K/100K 时反而比哈希表快。原因：固定 1021 槽位的哈希表在万级以上严重超载（链长 23-102），O(1) 均摊退化为 O(链长)；而 AVL 树高度仅从 14 增长到 17，O(log n) 保持稳定。
3. **哈希表瓶颈**：固定容量 1021 在 100K 时装载因子达 9794%，最长链 102。这充分说明固定容量哈希表不适合大规模数据，必须实现动态扩容（rehash）。
4. **AVL 平衡性**：100K 实际高度 17，理论最大值约 24，差距仅 7 层，旋转机制保持良好平衡。
5. **内存占用**：100K 时约 28MB，每条记录约 290 字节（含 AVLNode 和 HashNode 开销）。

### C.5 发现的 AVL 删除 Bug 及修复

在压力测试开发过程中，发现原有 `avl_delete` 函数的前驱替换策略存在一个隐蔽 bug：当删除双子节点时，前驱的 `data` 指针被转移后置为 NULL，但递归删除前驱时 `strcmp(key, root->data->key)` 访问了 NULL 指针，导致段错误。

**修复**：在 `avl_delete` 入口处增加 `data == NULL` 的守卫检查，遇到 NULL data 的节点直接按叶子/单子节点处理。该修复使删除功能在压力测试中稳定通过。

### D.1 问题描述

模拟内存仅能容纳 1000 条记录的极端场景，对 10 万条选课记录按学号进行外部排序。

### D.2 算法设计

采用**多路归并排序（External Merge Sort）**，分两阶段：

- **Phase 1 — 分片排序**：从输入文件逐行读取，每满 1000 条在内存中用快速排序（按学号），写入临时 chunk 文件 `tmp/chunk_XXXX_0.jsonl`。100K 条共产生 100 个 chunk 文件。
- **Phase 2 — 多路归并**：使用 k=20 路最小堆进行归并。Pass 1 将 100 个 chunk 归并为 5 个，Pass 2 将 5 个归并为 1 个最终输出文件。

### D.3 最小堆实现

```c
typedef struct {
    CourseRecord record;    // 值拷贝
    int chunk_index;        // 来源 chunk 编号
    int valid;
} HeapNode;
```

堆操作：`heap_push`（上滤）、`heap_pop`（下滤），按 `student_id` 字典序比较。

### D.5 复杂度分析

- **时间复杂度**：Phase 1 为 O(N log M)，Phase 2 为 O(N log k × passes) = O(N log N)。总计 O(N log N)。
- **I/O 复杂度**：理论值 2N × (1 + ⌈log_k(N/M)⌉) = 2 × 100K × (1 + 2) = 600,000 条次。实际 600,000 条次，与理论完全吻合。
- **每条记录平均被读写 6 次**（Phase 1 读 1 写 1，两轮归并各读 1 写 1）。
