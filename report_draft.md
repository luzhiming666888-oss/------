## 第 2 章 数据结构设计

### 2.1 逻辑结构设计

选课记录在逻辑上是一组**无序的扁平键值对集合**。本系统为满足不同操作场景的性能需求，将同一份数据同时映射到两种逻辑结构中，构成**双索引架构**：

- **AVL 平衡二叉搜索树**：以 `key`（学号\_课程号）为排序键，维护所有记录的有序性，支持 O(log n) 的精确增删查改及中序遍历（有序输出）。
- **拉链法哈希表**：以相同的 `key` 为哈希键，提供 O(1) 均摊的精确查找，用于高频精确检索场景。

两种结构共享同一块 `CourseRecord` 堆内存（通过指针引用，不进行数据拷贝），由 AVL 树节点负责内存的唯一所有权与最终释放，哈希表仅持有裸指针。

双索引架构示意如下：

```
+-----------+         +------------------+
|  AVL Tree |──data──▶|  CourseRecord    |◀──data──+  HashTable
|  (有序)   |         |  (唯一堆内存)    |         |  (快速检索)
+-----------+         +------------------+         +-----------+
```

### 2.2 ADT 定义

#### 2.2.1 CourseRecord（选课记录）

```c
/* 字段长度宏 */
#define STUDENT_ID_LEN   13   /* 学号 12 位 + '\0' */
#define NAME_LEN         32   /* 姓名（含中文 UTF-8）*/
#define COLLEGE_LEN      64   /* 学院名称 */
#define COURSE_ID_LEN    9    /* 课程编号 8 位 + '\0' */
#define COURSE_NAME_LEN  64   /* 课程名称 */
#define SEMESTER_LEN     16   /* 学期，格式 "2024-2025-1" */
#define DATE_LEN         11   /* 日期，格式 "YYYY-MM-DD" */
#define KEY_LEN          22   /* 主键 = 学号(12) + '_' + 课程号(8) + '\0' */

typedef struct CourseRecord {
    char  student_id[STUDENT_ID_LEN];   /* 学号（12位）         */
    char  name[NAME_LEN];               /* 学生姓名             */
    char  college[COLLEGE_LEN];         /* 所属学院             */
    char  course_id[COURSE_ID_LEN];     /* 课程编号（8位）      */
    char  course_name[COURSE_NAME_LEN]; /* 课程名称             */
    float credit;                       /* 学分                 */
    char  semester[SEMESTER_LEN];       /* 选课学期             */
    char  date[DATE_LEN];               /* 选课日期             */
    int   score;                        /* 成绩（0-100，-1未录）*/
    char  key[KEY_LEN];                 /* 唯一主键（自动生成） */
} CourseRecord;
```

**ADT 操作集合：**

| 操作名称             | 原型                                                       | 语义                         |
| -------------------- | ---------------------------------------------------------- | ---------------------------- |
| 生成主键             | `void record_make_key(CourseRecord *r)`                    | 由 student_id + course_id 拼接 |
| 格式化打印           | `void record_print(const CourseRecord *r)`                 | 将记录格式化输出到标准输出   |
| 主键比较             | `int record_compare_key(const CourseRecord *a, const CourseRecord *b)` | 返回字典序比较结果 |

#### 2.2.2 AVLNode（AVL 树节点）

```c
typedef struct AVLNode {
    CourseRecord  *data;        /* 记录指针（唯一所有权）*/
    int            height;      /* 子树高度（叶节点 = 1）*/
    struct AVLNode *left;       /* 左子节点              */
    struct AVLNode *right;      /* 右子节点              */
} AVLNode;
```

| 操作名称   | 原型                                                        | 时间复杂度 |
| ---------- | ----------------------------------------------------------- | :--------: |

| 插入节点   | `AVLNode *avl_insert(AVLNode *root, CourseRecord *record)`  | O(log n)   |

#### 2.2.3 HashTable（拉链法哈希表）

```c
typedef struct HashNode {
    CourseRecord    *data;  /* 记录指针（非所有权）*/
    struct HashNode *next;  /* 拉链下一节点        */
} HashNode;

typedef struct HashTable {
    HashNode **buckets;     /* 槽位数组            */
    int        capacity;    /* 槽位总数（质数）    */
    int        size;        /* 当前记录总数        */
} HashTable;
```

**哈希表 ADT 操作集合：**

| 操作名称   | 原型                                                              | 均摊复杂度 |
| ---------- | ----------------------------------------------------------------- | :--------: |
| 初始化     | `int hash_init(HashTable *ht, int capacity)`                      | O(m)       |
| 插入       | `int hash_insert(HashTable *ht, CourseRecord *record)`            | O(1)       |
| 删除       | `int hash_delete(HashTable *ht, const char *key)`                 | O(1)       |
| 精确查找   | `CourseRecord *hash_search(HashTable *ht, const char *key)`       | O(1)       |
| 更新指针   | `int hash_update(HashTable *ht, const char *key, CourseRecord *r)`| O(1)       |
| 销毁       | `void hash_destroy(HashTable *ht)`                                | O(m+n)     |

> *m = 槽位数，n = 记录数*

> **【撰写说明】**
>
> ADT 定义部分已基本完整，需补充：
>
> 1. **哈希函数说明**：本项目采用 FNV-1a（Fowler-Noll-Vo 1a）哈希算法，其核心步骤为 `hash ^= byte; hash *= 16777619`，初始偏移量 `2166136261`。相比备选的 DJB2 算法（`hash * 33 ^ c`，初始值 5381），FNV-1a 在短字符串场景下分布更均匀、碰撞率更低，尤其适合本项目"12 位学号\_8 位课程号"的主键格式。
> 2. **平衡因子定义**：AVL 树平衡因子 BF = 左子高 - 右子高，BF ∈ {-1, 0, 1} 为合法状态，|BF| = 2 时触发旋转修复。
> 3. **内存所有权说明**：`CourseRecord` 堆内存唯一所有权在 AVL 树节点，哈希表的 `HashNode.data` 为裸指针共享，`hash_destroy()` 仅释放 `HashNode` 链表节点本身，不 `free(data)`，防止双重释放错误。

### 2.3 存储结构方案

#### 2.3.1 内存分配方式

本系统采用**动态堆分配**策略：

- 每条 `CourseRecord` 在插入时通过 `malloc(sizeof(CourseRecord))` 分配，生命周期由对应 AVL 节点管理。
- 每个 `AVLNode` 同样通过 `malloc(sizeof(AVLNode))` 分配，`avl_destroy()` 负责后序遍历释放。
- `HashNode` 链表节点通过 `malloc(sizeof(HashNode))` 分配，`hash_destroy()` 负责逐槽释放。

#### 2.3.2 空间开销估算

| 结构         | 单节点字节数                          | n = 10,000 时总占用    |
| ------------ | ------------------------------------- | ---------------------- |
| CourseRecord | ≈ 13+32+64+9+64+4+16+11+4+22 = 239 B，对齐后约 **240 B** | ≈ 2.4 MB |
| AVLNode      | 8（指针）+ 4（height）+ 8+8（left/right）= **28 B**（64位系统） | ≈ 0.27 MB |
| HashNode     | 8（data指针）+ 8（next指针）= **16 B** | ≈ 0.15 MB |
| HashTable buckets 数组 | 1021 × 8 B = 约 **8 KB**   | 固定                   |
| **合计**     |                                       | **≈ 2.83 MB**          |

#### 2.3.3 文件持久化格式

数据文件采用 **JSON Lines** 格式（每行一个独立 JSON 对象），不依赖任何外部解析库，序列化与反序列化均为手写实现。示例行如下：

```json
{"sid":"202401011234","name":"张三","college":"计算机学院","cid":"CS300102","cname":"数据结构","credit":3.0,"semester":"2024-2025-1","date":"2024-09-01","score":88}
```

> **【撰写说明】**
>
> 1. **为何选 JSON Lines 而非 CSV**：JSON Lines 自描述性更强，字段顺序无关，新增字段时对旧文件有良好的前向兼容性；CSV 在含逗号的字段（如课程名称含英文逗号）时需要额外的转义处理，增加解析复杂度。
> 2. **手写解析器的核心思路**（一段话）：扫描字符串，找到 `"key":` 模式后提取引号内的值（字符串字段）或直接读取数字（数值字段），遇到非法格式返回 -1 报错。
> 3. **数据文件路径约定**：运行目录下的 `data/records.jsonl`，程序启动时自动创建（若不存在）。

## 第 3 章 方案对比与选型论证

### 3.1 候选方案介绍

在进行技术选型时，本项目重点考察了以下三种数据结构方案：

**方案一：双向链表（Doubly Linked List）**

双向链表是最直观的线性动态结构，插入和删除操作在已知节点位置时为 O(1)，但由于缺乏索引，按键查找必须从头遍历，时间复杂度为 O(n)，在大规模数据场景下性能不可接受。

**方案二：哈希表（Hash Table，拉链法）**

哈希表通过散列函数将键映射到桶槽，实现均摊 O(1) 的精确查找。其缺点是：① 无法保持键的有序性，不支持范围查询和有序遍历；② 哈希碰撞严重时性能退化；③ 动态扩容实现复杂。

**方案三：AVL 平衡二叉搜索树（AVL Tree）**

AVL 树是一种自平衡的 BST，任意节点左右子树高度差不超过 1，保证最坏情况下增删查均为 O(log n)。相较于普通 BST，AVL 树避免了退化为链表的极端情况；相较于红黑树，AVL 树的平衡条件更严格，查询性能略优（树高更低），但旋转次数略多。

---

### 3.2 多维度对比分析

| 对比维度           | 双向链表      | 哈希表（拉链）         | AVL 树              |
| ------------------ | :-----------: | :--------------------: | :-----------------: |
| **精确查找**       | O(n)          | O(1) 均摊              | O(log n)            |
| **插入**           | O(1)（尾插） | O(1) 均摊              | O(log n)            |
| **删除**           | O(n)（需查找）| O(1) 均摊              | O(log n)            |
| **有序遍历**       | O(n)（有序需额外排序）| ✗ 不支持原生有序 | O(n)（中序天然有序）|
| **范围查询**       | O(n)          | ✗ 不支持               | O(log n + k)        |
| **最坏情况查找**   | O(n)          | O(n)（退化时）         | O(log n)            |
| **空间开销**       | 低            | 中（桶数组 + 链表节点）| 中（节点含 height） |
| **实现复杂度**     | 低            | 中                     | 高（旋转逻辑复杂）  |
| **内存局部性**     | 差            | 一般                   | 一般                |
| **适用主要场景**   | 有序链式操作  | 高频精确检索           | 增删查改 + 有序输出 |

> "从表中可见，单一哈希表虽然精确查找最快，但无法支持有序遍历和范围查询；而 AVL 树在各操作均能保证 O(log n) 的稳定性能，且天然支持有序输出。两者各有侧重，因此本项目采用双索引架构，以哈希表承担高频精确检索，以 AVL 树承担有序遍历与范围操作。"

### 3.3 选型结论

综合 3.2 节的对比分析，本项目最终采用 **"AVL 树 + 哈希表"双索引架构**，并使用**手写极简 JSON 序列化/反序列化**方案，工程合理性论证如下：

**（1）双索引优势互补**

AVL 树与哈希表在查询模式上天然互补：哈希表以 O(1) 均摊复杂度处理高频精确查找（如"按学号\_课程号检索单条记录"），AVL 树以 O(n) 中序遍历支持全集有序输出（如"按成绩排序展示所有记录"）和 O(log n) 的增删维护。两者共享同一块 `CourseRecord` 堆内存，不产生数据冗余。

**（2）AVL 树优于普通 BST 与跳表**

普通 BST 在数据按主键顺序插入时退化为链表，查询退化为 O(n)，在校园系统中学号通常按序递增，此退化场景极易触发。跳表虽然实现相对简单，但随机层数引入不确定性，且在 C 语言中需要手动管理多层指针，维护成本高于 AVL 树。

**（3）手写 JSON 满足工程约束**

课程设计明确禁止使用第三方库（`cJSON`、`jansson` 等），而 JSON Lines 格式每行结构固定、字段有限，手写扫描解析器代码量约 80～120 行，完全可控。相较于自定义二进制格式，JSON Lines 具有人类可读性，便于调试和数据导入/导出。

### 4.2 核心函数说明

#### 4.2.1 哈希精确检索：`hash_search()`

**函数原型：**

```c
CourseRecord *hash_search(HashTable *ht, const char *key);
```

**功能描述：**

在哈希表中按唯一主键（学号\_课程号）进行 O(1) 均摊的精确查找。调用者传入 `HashTable` 指针和 key 字符串，函数返回匹配的 `CourseRecord` 指针（未找到返回 NULL）。该函数是双索引架构中"高频精确检索"的主要入口，例如修改记录前先用哈希表快速定位目标，再通过 AVL 树维护有序性。

**实现思路：**

1. 调用 `hash_fnv1a(key)` 计算散列值，对 `ht->capacity` 取模得到槽位下标 `idx`。
2. 从 `ht->buckets[idx]` 开始遍历拉链，逐节点比较 `node->data->key` 与 `key`。
3. 找到则返回 `node->data`，遍历完未找到则返回 `NULL`。

**时间复杂度：** 均摊 O(1)，最坏 O(n)（全槽碰撞）

#### 4.2.2 AVL 树插入与旋转：`avl_insert()` / `avl_rotate_*()`

**函数原型：**

```c
AVLNode *avl_insert(AVLNode *root, CourseRecord *record);
AVLNode *avl_rotate_ll(AVLNode *z);   /* LL 型右旋 */
AVLNode *avl_rotate_rr(AVLNode *z);   /* RR 型左旋 */
AVLNode *avl_rotate_lr(AVLNode *z);   /* LR 型双旋 */
AVLNode *avl_rotate_rl(AVLNode *z);   /* RL 型双旋 */
```

**插入流程说明：**

1. 按 BST 规则递归向下定位插入位置，`strcmp` 比较新记录的 `key` 与当前节点 `data->key`，小于则走向左子树，大于则走向右子树，等于则拒绝插入（主键唯一性约束）。
2. 递归到空位时，调用 `avl_new_node(record)` 创建新叶节点（`height = 1`），挂接到父节点的对应子指针上。
3. 递归返回过程中，逐层调用 `AVL_UPDATE_HEIGHT(node)` 宏更新高度（取左右子树最大高度 + 1）。
4. 计算当前节点平衡因子 BF，若 |BF| = 2 则判断失衡类型并选择对应旋转：
   - BF = 2 且左子 BF ≥ 0 → LL 型（右旋 `avl_rotate_ll`）
   - BF = 2 且左子 BF < 0 → LR 型（先对左子左旋，再对当前节点右旋 `avl_rotate_lr`）
   - BF = -2 且右子 BF ≤ 0 → RR 型（左旋 `avl_rotate_rr`）
   - BF = -2 且右子 BF > 0 → RL 型（先对右子右旋，再对当前节点左旋 `avl_rotate_rl`）
5. 旋转后更新涉及节点的高度，返回新子树根节点。

**删除流程说明：**

1. 按 BST 规则递归查找待删除节点，`strcmp` 比较 `key` 定位目标。
2. 找到目标节点后分三种情况处理：
   - **叶子节点（无子节点）**：直接 `free(data)` 和 `free(node)`，返回 NULL。
   - **单子节点**：用子节点替换当前节点，释放当前节点的 `data` 和 `node` 本身，返回子节点指针。
   - **双子节点**：采用"前驱替换"策略——找到左子树的最大节点（前驱），将其 `data` 指针转移给当前节点（先释放当前节点的旧 `data`），然后将前驱节点的 `data` 置 NULL（防止双重释放），最后递归删除前驱节点（此时前驱至多有一个左子）。
3. 递归返回过程中，同样逐层更新高度并执行 `avl_rebalance` 平衡修复。

### 4.3 关键代码片段

#### 4.3.1 FNV-1a 哈希函数

```c
unsigned long hash_fnv1a(const char *key) {
    unsigned long hash = 2166136261UL;
    const unsigned long FNV_PRIME = 16777619UL;
    while (*key) {
        hash ^= (unsigned char)*key++;
        hash *= FNV_PRIME;
    }
    return hash;
}
```

FNV-1a 算法由 Glenn Fowler、Landon Curt Noll 和 Phong Vo 共同设计。其核心思想是对每个字节先执行 XOR 异或操作再乘以质数 `16777619`（FNV prime），初始偏移量 `2166136261`（FNV offset basis）是经验值。相比 DJB2 的 `hash * 33 ^ c`，FNV-1a 先 XOR 后乘的策略能更好地混淆高低位，在短字符串场景下碰撞率更低、分布更均匀，尤其适合本项目 22 字符长度的主键（学号\_课程号）。

#### 4.3.2 AVL 旋转核心代码

以 LL 型右旋为例，展示指针操作与高度更新的完整实现：

```c
/**
 * @brief   LL 型失衡修复 —— 右旋
 * z 的左子树的左子树插入导致失衡。
 * 将 z 的左子 l 提升为新子树根，z 成为 l 的右子。
 *
 * 旋转前:        z               旋转后:      l
 *               / \                         /   \
 *              l   T4                     T1     z
 *             / \                                / \
 *            T1  T2                            T2  T4
 */
AVLNode *avl_rotate_ll(AVLNode *z) {
    AVLNode *l = z->left;     /* l 为 z 的左子节点，旋转后成为新根 */

    z->left   = l->right;     /* l 的右子树 T2 挂到 z 的左侧 */
    l->right  = z;            /* z 成为 l 的右子 */

    AVL_UPDATE_HEIGHT(z);     /* 先更新下层节点 z 的高度 */
    AVL_UPDATE_HEIGHT(l);     /* 再更新新根 l 的高度 */

    return l;                 /* 返回新子树根 */
}
```

**关键点分析：**

1. **指针操作顺序**：必须先将 `l->right` 赋给 `z->left`（保存 T2 子树），再将 `z` 赋给 `l->right`，顺序不可颠倒，否则 `l->right` 原有的引用会丢失。
2. **高度更新顺序**：先更新处于下层的 `z`（其子树已确定），再更新上层的 `l`（依赖 `z` 的新高度），保证 `AVL_UPDATE_HEIGHT` 宏读取到的子树高度是最新的。
3. **平衡修复函数 `avl_rebalance`**：在插入/删除的递归回溯路径上，对每个节点计算 `avl_balance_factor`，根据 BF 值和子节点 BF 的符号判断 LL/RR/LR/RL 四种失衡类型，调用对应旋转函数。

#### 4.3.3 JSON Lines 反序列化核心代码

```c
/**
 * @brief  在 json 字符串中查找 "field" 后面的值，写入 out_buf
 * @return 0 成功；-1 未找到该字段
 *
 * 支持的值类型：字符串 ("...") 和整数/浮点数（裸数字）
 */
static int json_extract(const char *json, const char *field,
                        char *out_buf, int out_sz) {
    /* 构造搜索模式："field" */
    char pattern[64];
    snprintf(pattern, sizeof(pattern), "\"%s\"", field);

    const char *p = strstr(json, pattern);
    if (!p) return -1;

    /* 跳过 "field" 部分 */
    p += strlen(pattern);

    /* 跳过空白和冒号 */
    while (*p && (isspace((unsigned char)*p) || *p == ':')) p++;
    if (!*p) return -1;

    if (*p == '"') {
        /* 字符串值：提取到下一个未转义的 '"' */
        p++;
        int i = 0;
        while (*p && *p != '"' && i < out_sz - 1) {
            if (*p == '\\' && *(p + 1) == '"') {
                if (i < out_sz - 1) out_buf[i++] = '"';
                p += 2;
            } else {
                out_buf[i++] = *p++;
            }
        }
        out_buf[i] = '\0';
        return 0;
    } else {
        /* 数值类型：提取连续的数字、小数点、负号 */
        int i = 0;
        while (*p && !isspace((unsigned char)*p) && *p != ',' && *p != '}' && i < out_sz - 1) {
            out_buf[i++] = *p++;
        }
        out_buf[i] = '\0';
        return 0;
    }
}

int json_to_record(const char *json, CourseRecord *out) {
    if (!json || !out) return -1;
    memset(out, 0, sizeof(CourseRecord));

    char buf[256];

    /* 逐字段提取，任一字段缺失则返回 -1 */
    if (json_extract(json, "sid", buf, sizeof(buf)) != 0) return -1;
    util_strncpy(out->student_id, buf, STUDENT_ID_LEN);

    if (json_extract(json, "name", buf, sizeof(buf)) != 0) return -1;
    util_strncpy(out->name, buf, NAME_LEN);
    /* ... 其余字段同理 ... */

    /* 自动生成唯一主键 */
    record_make_key(out);
    return 0;
}
```

**关键点分析：**

1. **字段顺序无关**：`json_extract` 通过 `strstr` 搜索 `"field"` 模式定位字段，不依赖 JSON 中键值对的顺序，符合 JSON 规范。
2. **转义处理**：字符串值中支持 `\"` 转义引号，虽然本项目测试数据不含转义字符，但代码已预留处理逻辑。
3. **数值与字符串分流**：通过判断值的首字符是否为 `"` 区分字符串字段与数值字段，分别走不同的提取分支。
4. **安全截断**：所有字段提取均通过 `util_strncpy` 写入目标缓冲区，保证以 `\0` 结尾，防止缓冲区溢出。

#### 4.3.4 CLI 菜单交互核心代码

`main.c` 采用**统一 `fgets` 输入策略**，避免 `scanf` 与 `fgets` 混用导致的 stdin 缓冲区错位问题：

```c
/**
 * @brief  读取一行字符串（去除首尾空白，自动消耗溢出字符）
 */
static void read_line(char *buf, int size) {
    if (!fgets(buf, size, stdin)) {
        buf[0] = '\0';
        return;
    }

    /* 检查 fgets 是否读到了完整的一行（以 \n 结尾） */
    int len = (int)strlen(buf);
    int has_newline = (len > 0 && buf[len - 1] == '\n');

    /* 如果没读到换行符，说明该行还有剩余字符在 stdin 中，
       需要消耗掉，避免污染下一次读取 */
    if (!has_newline) {
        int c;
        while ((c = getchar()) != '\n' && c != EOF) { }
    }

    /* 去除首尾空白（包括 \r\n） */
    util_trim(buf);
}
```

**主菜单循环结构：**

```c
static void menu_main(void) {
    char buf[16];
    while (1) {
        /* 打印菜单 ... */
        printf("请输入选项：");
        read_line(buf, sizeof(buf));

        if (feof(stdin)) {
            printf("\n[INFO] 检测到输入结束，自动退出。\n");
            return;
        }

        int choice = atoi(buf);
        switch (choice) {
            case 1: menu_insert(); break;
            case 2: menu_delete(); break;
            case 3: menu_search(); break;
            case 4: menu_update(); break;
            case 5: menu_list();   break;
            case 6: menu_stats();  break;
            case 0: return;
            default: printf("[WARN] 无效选项。\n"); break;
        }
    }
}
```

**关键设计决策：**

1. **统一 `fgets` + `atoi`/`atof`**：所有输入均通过 `read_line` 读取字符串后转换，彻底消除 `scanf` 留残 `\n` 导致 `fgets` 读到空行的经典 C 语言 bug。
2. **缓冲区溢出保护**：当 `fgets` 的 `size` 小于输入行长度时（如 `STUDENT_ID_LEN=13` 读 12 字符学号），`read_line` 自动消耗 stdin 中剩余字符，保证下一次读取对齐。
3. **EOF 优雅退出**：主循环检测 `feof(stdin)`，在管道输入耗尽时自动退出，避免无限循环。
4. **修改操作的原地更新策略**：修改记录时仅更新 `CourseRecord` 字段值（学号和课程编号不可改），由于 AVL 树和哈希表都持有同一指针，无需重建索引。
